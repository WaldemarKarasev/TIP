# Лекция 4. Flow

## `Liveness analysis` (слайд 6)

Для `liveness analysis` берем булеан над множеством всех переменных программы:

L = P(Vars)

Элемент решётки - это множество переменных, которые считаются живыми в данной точке программы. Порядок задаётся включением множеств:

$A \sqsubseteq B \iff A \subseteq B$

То есть множество $A$ меньше или равно множеству $B$, если все переменные из $A$ содержатся в $B$.

Тогда:

$\bot = \varnothing$

$\top = Vars$

$A \sqcup B = A \cup B$

$A \sqcap B = A \cap B$

Такой анализ является `may`-анализом. Если переменная входит в результат, она возможно жива. Если переменной нет в результате, то она точно мертва. Это безопасно для оптимизаций, потому что нас интересуют переменные, которые можно гарантированно считать мертвыми.

## Алгоритмическая сложность структурного алгоритма (слайд 15)

Псевдокод для структурного алгоритма:
```
for v in nodes:
    live[v] = {}
    worklist.push(v)

while worklist not empty:
    v = worklist.pop()

    old = live[v]
    new = recompute(v)

    if new != old:
        live[v] = new

        for p in predecessors(v):
            worklist.push(p)
```
Пересчёт вершины:
```
function recompute(v):
    result = {}

    for s in successors(v):
        result = result ∪ live[s]

    result = transfer(v, result)

    return result
```
Один пересчёт стоит O(k), потому что состояние вершины - множество живых переменных размера до k, и операции union, difference, сравнение множеств работают за O(k).

Одна вершина может изменить своё значение максимум k раз, так как в множество живых переменных можно добавить не более k переменных.

Изменения распространяются по CFG:

$O(k · n)$ - максимум изменений значений live[v] по всем вершинам
$O(k · m)$ - максимум распространений этих изменений по рёбрам CFG

Итого количество распространений:

$O(k · (n + m))$

Каждое распространение требует пересчёта множества за O(k), поэтому итоговая сложность:

$O(k^2 · (n + m))$

Если CFG разреженный, то m = O(n), поэтому:

$O(k^2 · (n + m)) = O(k^2 · n)$

Если в CFG нет циклов, то для backward-анализа достаточно одного прохода в обратном топологическом порядке. Тогда каждая вершина и каждое ребро обрабатываются один раз, поэтому сложность равна O(k * (n + m)).

По памяти нужно хранить множество живых переменных для каждой вершины CFG. Каждое множество имеет размер до k, всего вершин n, поэтому основная сложность по памяти равна O(n * k). Дополнительно хранится сам CFG размера O(n + m).

### Какую решетку брать для `Available expressions analysis`? (слайды 18–19)

`Available expressions analysis` определяет выражения, которые гарантированно уже были вычислены до данной точки программы. Это must-анализ

Если `E` - множество всех рассматриваемых выражений. Элемент решетки - множество выражений, которые считаются готовыми в данной точке программы.

Для available expressions:

$\top = \varnothing$

$\bot = E$

$A \sqcup B = A \cap B$

То есть join задается пересечением, потому что на слиянии путей выражение остается готовым только в том случае, если оно готово на всех входящих путях.

Обычный булеан с join как union здесь не подходит: он мог бы посчитать выражение готовым, если оно вычислялось хотя бы на одном пути, а для available expressions нужна гарантия вычисления на всех путях.

## Какую решетку брать для `Very busy expressions analysis`? (слайд 28)
Для very busy expressions берем перевернутый булеан над множеством всех нетривиальных выражений программы. Элемент решетки - это множество выражений, которые считаются очень занятыми в данной точке.

Это must-анализ: выражение можно считать очень занятым только если оно будет использовано в будущем на всех путях выполнения. Поэтому на слиянии путей используется пересечение, а не объединение:

JOIN(v) = intersection по successors(v)

Если выражение есть не во всех будущих ветках, то его нельзя считать очень занятым. Анализ является backward-анализом, потому что свойство зависит от будущих использований выражения.

## Система ограничений и ее решение (слайд 30)

Рассмотрим программу:

```tip
var x, a, b;

x = input;
a = x - 1;
b = x - 2;

while (x > 0) {
    output a * b - x;
    x = x - 1;
}

output a * b;
```

Множество рассматриваемых выражений:

```text
Exprs = {x - 1, x - 2, a * b, a * b - x}
```

Используем правила:

```text
[[x = E]] = removerefs(JOIN(n), x) ∪ exprs(n)

[[n]] = JOIN(n) ∪ exprs(n)     для остальных узлов

JOIN(v) = intersection по всем succ(v)
```

где `removerefs(S, x)` удаляет из множества `S` все выражения, в которых встречается переменная `x`.

#### Система ограничений

$[[var\ x,a,b]] = [[x = input]]$

$[[x = input]] = removerefs([[a = x - 1]], x)$

$[[a = x - 1]] = removerefs([[b = x - 2]], a) \cup {x - 1}$

$[[b = x - 2]] = removerefs([[x > 0]], b) \cup {x - 2}$

$[[x > 0]] = [[output\ a * b - x]] \cap [[output\ a * b]]$

$[[output\ a * b - x]] = [[x = x - 1]] \cup {a * b, a * b - x}$

$[[x = x - 1]] = removerefs([[x > 0]], x) \cup {x - 1}$

$[[output\ a * b]] = {a * b}$


#### Решение

$[[var x,a,b]] = \{\}$

$[[x = input]] = \{\}$

$[[a = x - 1]] = \{x - 1, x - 2\}$

$[[b = x - 2]] = \{x - 2\}$

$[[x > 0]] = \{a * b\}$

$[[output\ a * b - x]] = \{x - 1, a * b, a * b - x\}$

$[[x = x - 1]] = \{x - 1, a * b\}$

$[[output\ a * b]] = \{a * b\}$

#### Вывод

Выражение `a * b` является very busy перед условием `x > 0`, потому что из этой точки оно будет использовано на любом дальнейшем пути выполнения:

Если цикл продолжается, используется output a * b - x;
Если цикл завершается, используется output a * b.

Поэтому выражение `a * b` можно заранее вычислить перед проверкой условия цикла.

## Какую решётку брать для `Reaching definitions analysis`? (слайд 32)

`Reaching definitions analysis` определяет, какие присваивания или объявления могут отвечать за текущие значения переменных в данной точке программы. Это may-анализ.

Для анализа берём булеан над множеством всех определений программы:

Defs = множество всех объявлений и присваиваний
L = P(Defs)

Элемент решётки - множество определений, которые могут достигать данной точки программы.

Порядок задаётся включением множеств:

$ A \sqsubseteq B \iff A \subseteq B  $

Тогда:

$ \bot = \empty $
$ \top = Defs $
$ A \sqcup B = A \cup B $
$ A \sqcap B = A \cap B $

Join равен объединению, потому что определение должно попасть в результат, если оно может достигать данной точки хотя бы по одному пути CFG.

Например:

```tip
if (input) {
    x = 1;
} else {
    x = 2;
}

return x;
```

Перед `return x` возможны оба определения:

```text
{x = 1, x = 2}
```

потому что каждое из них достигает этой точки по одному из путей выполнения.

## `Constant propagation` и `Interval lattice` (слайды 40-45)

`Constant propagation` - это анализ, который определяет, какие значения переменных являются константами и могут быть вычислены на этапе компиляции.

Для простого `constant propagation` можно использовать решётку отображений:

$Vars \to FlatInt$

где $Vars$ - множество переменных программы, а $FlatInt$ - плоская решётка над множеством целых чисел.

Элементы $FlatInt$ имеют следующий смысл:

$\bot$ - информации ещё нет или состояние недостижимо;
$n \in \mathbb{Z}$ - переменная точно равна конкретной константе $n$;
$\top$ или $? $ - значение неизвестно или не является одной конкретной константой.

Например, состояние анализа может выглядеть так:

$x \mapsto 42$

$y \mapsto \top$

$z \mapsto \bot$

Такой анализ хорошо находит точные константы, но у него есть проблема: он не хранит более общую информацию о значениях. Например, если известно, что $x$ положительный, но неизвестно точное значение $x$, такая решётка не сможет доказать, что выражение $x > 0$ является константно истинным.

Чтобы решить эту проблему, можно попытаться смешать решётку констант и решётку знаков. Тогда анализ мог бы хранить не только точные значения вроде $x = 5$, но и информацию вида $x > 0$. Однако такая решётка быстро становится сложной: нужно определять порядок, join, meet и операции над элементами разных видов, например для выражений вида $1 + \top$.

Более общий подход - использовать интервальную решётку. В ней значение переменной описывается интервалом возможных значений:

$[3, 3]$ - точно $3$;
$[1, +\infty]$ - положительное значение;
$[0, +\infty]$ - неотрицательное значение;
$[-\infty, +\infty]$ - неизвестное значение;
$\bot$ - невозможное или недостижимое состояние.

Интервальная решётка обобщает `constant propagation` и `sign analysis`: точная константа $n$ представляется интервалом $[n, n]$, а информация о знаке - более широким интервалом, например $[1, +\infty]$.

Главная проблема `interval lattice` в том, что она бесконечна. Например, существует бесконечная возрастающая цепочка:

$[0, 0] \sqsubseteq [0, 1] \sqsubseteq [0, 2] \sqsubseteq [0, 3] \sqsubseteq \dots$

Из-за этого обычные алгоритмы поиска неподвижной точки, рассчитанные на решётки конечной высоты, могут не завершиться. Даже если учитывать ограниченность машинного `int`, решётка остаётся слишком большой для наивного анализа. Поэтому для интервального анализа обычно нужны дополнительные техники, например `widening`.
