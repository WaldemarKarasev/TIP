# Лекция 5. Intervals

## Арифметика и сравнения интервалов (слайды 6-7)

#### Сложность в арифметике интервалов?

Для арифметических операций общая формула:

$[l_1, h_1]\ op\ [l_2, h_2] = [\min(x\ op\ y), \max(x\ op\ y)]$

где $x \in [l_1, h_1]$, $y \in [l_2, h_2]$.

Если вычислять это напрямую, нужно перебрать все пары значений из двух интервалов. Если размеры интервалов равны $m$ и $n$, то сложность равна $O(m \cdot n)$. Для больших или бесконечных интервалов такой подход неприменим.

#### Почему нельзя сделать арифметику интервалов проще?

Нельзя использовать одну простую формулу вида:

$[l_1, h_1]\ op\ [l_2, h_2] = [l_1\ op\ l_2, h_1\ op\ h_2]$

потому что для некоторых операций она неверна.

Например:

$[0, 10] - [0, 10] \neq [0 - 0, 10 - 10] = [0, 0]$

Правильно:

$[0, 10] - [0, 10] = [-10, 10]$

Для умножения тоже нужно учитывать разные комбинации границ:

$[-2, 3] \cdot [-4, 5] = [-12, 15]$

потому что минимум и максимум получаются из разных комбинаций концов интервалов.

#### Почему для сравнений нельзя всегда возвращать только $[0, 1]$?

Для сравнений самый простой корректный вариант — всегда возвращать $[0, 1]$, потому что результат сравнения может быть либо ложным, либо истинным. Но это слишком неточно.

Можно сделать точнее:

если сравнение истинно для всех значений из интервалов, возвращаем $[1, 1]$
если сравнение ложно для всех значений из интервалов, возвращаем $[0, 0]$
если возможны оба результата, возвращаем $[0, 1]$

Примеры:

$[1, 3] < [5, 7] = [1, 1]$

$[5, 7] < [1, 3] = [0, 0]$

$[1, 5] < [3, 7] = [0, 1]$

#### Почему для сравнений нельзя сделать “как выше”?

Для сравнений нельзя просто применять арифметический подход по границам. Например:

$[1, 3] == [2, 2] = [0, 1]$

потому что внутри первого интервала есть значение $2$, значит равенство может быть истинным, но не обязательно истинно всегда.

## Почему интервальный анализ может расходиться? (слайд 8)

#### Почему обычный алгоритм неподвижной точки не работает для бесконечной интервальной решётки?

Обычный алгоритм поиска неподвижной точки гарантированно завершается, если решётка имеет конечную высоту. У интервальной решётки высота бесконечна, потому что в ней существуют бесконечные возрастающие цепочки:

$[0,0] \sqsubseteq [0,1] \sqsubseteq [0,2] \sqsubseteq [0,3] \sqsubseteq \dots$

Если на каждой итерации анализа интервал становится всё шире, алгоритм может бесконечно уточнять результат и не дойти до неподвижной точки за конечное число шагов.

#### Пример программы на TIP, где анализ расходится

```tip
var x;

x = 0;

while (input) {
    x = x + 1;
}
```

После инициализации известно:

$x \mapsto [0,0]$

После одной итерации цикла:

$x \mapsto [0,1]$

После двух итераций:

$x \mapsto [0,2]$

После трёх итераций:

$x \mapsto [0,3]$

И так далее:

$x \mapsto [0,n]$

Так как цикл может выполниться любое число раз, анализ строит бесконечную цепочку:

$[0,0] \sqsubseteq [0,1] \sqsubseteq [0,2] \sqsubseteq [0,3] \sqsubseteq \dots$

Поэтому без дополнительных техник анализ может не завершиться.

#### Почему нельзя просто ограничить размер целых чисел?

Можно сказать, что реальные `int` имеют конечный размер, например $32$ бита. Тогда формально множество возможных значений конечно.

Но на практике это не решает проблему полностью:

* решётка всё равно остаётся слишком большой;
* алгоритм может потребовать огромное число итераций;
* анализатор становится непрактичным по времени;
* в языке могут быть не только `int`, но и `bigint`;
* при переполнениях нужно отдельно учитывать семантику языка.

Например, если переменная растёт от $0$ до $2^{31}-1$, то формально анализ завершится, но может потребовать слишком много шагов.

#### Что нужно менять?

Нужно не ждать, пока анализ сам дойдёт до точной неподвижной точки, а специально загрублять результат. Для этого используется `widening`.

Идея `widening` в том, чтобы заменить бесконечную возрастающую цепочку более грубым, но конечным приближением. Например, вместо последовательности:

$[0,0] \sqsubseteq [0,1] \sqsubseteq [0,2] \sqsubseteq [0,3] \sqsubseteq \dots$

можно перейти к приближению:

$[0,+\infty]$

Такой результат менее точный, но безопасный: он всё ещё содержит все возможные значения переменной и позволяет анализу завершиться.

## Какую конечную версию интервальной решётки можно придумать? (слайд 12)

#### Как сделать интервальную решётку конечной?

Бесконечная интервальная решётка содержит интервалы с произвольными границами, например:

$[0,0], [0,1], [0,2], [0,3], \dots$

Из-за этого алгоритм поиска неподвижной точки может не завершиться.

Чтобы получить конечную версию решётки, можно заранее выбрать конечное множество допустимых границ $B$ и разрешить только интервалы, границы которых принадлежат этому множеству.

Например:

$B = {-\infty, 0, 1, 7, +\infty}$

Тогда вместо произвольных интервалов мы используем только интервалы вида:

$[b_1, b_2]$

где $b_1 \in B$, $b_2 \in B$, $b_1 \le b_2$.

#### Как загрублять интервалы?

Если анализ получает интервал $[a,b]$, границы которого не входят в $B$, то его нужно расширить до ближайшего интервала с границами из $B$.

Идея такая:

* левую границу округляем вниз до ближайшего элемента из $B$;
* правую границу округляем вверх до ближайшего элемента из $B$.

Например, если:

$B = {-\infty, 0, 1, 7, +\infty}$

то:

$[2,5] \mapsto [1,7]$

$[8,8] \mapsto [7,+\infty]$

$[-3,4] \mapsto [-\infty,7]$

Такое загрубление безопасно, потому что новый интервал содержит все значения старого интервала.

#### Как выбрать множество $B$?

Один естественный вариант — взять константы, которые встречаются в программе, и добавить к ним $-\infty$ и $+\infty$.

Например, для программы:

```tip
var x,y;
y = 0;
while (input) {
    x = 7;
    x = x + 1;
    y = y + 1;
}
```

можно взять:

$B = {-\infty, 0, 1, 7, +\infty}$

Здесь $0$, $1$ и $7$ — константы из программы.

#### Любое ли множество $B$ подходит?

Формально $B$ должно быть конечным и должно позволять безопасно расширять любой интервал. Поэтому практически важно, чтобы в $B$ были $-\infty$ и $+\infty$.

Если не добавить бесконечности, то для некоторых интервалов нельзя будет корректно найти ближайшую внешнюю границу. Например, если $B = {0,1,7}$, то интервал $[8,8]$ уже нельзя безопасно округлить вверх внутри $B$.

Поэтому лучше использовать множество вида:

$B = {-\infty} \cup C \cup {+\infty}$

где $C$ — конечное множество выбранных чисел, например констант из программы.

#### Что мы получаем?

В результате бесконечная цепочка:

$[0,0] \sqsubseteq [0,1] \sqsubseteq [0,2] \sqsubseteq [0,3] \sqsubseteq \dots$

заменяется конечной цепочкой по выбранным границам. Например при:

$B = {-\infty, 0, 1, 7, +\infty}$

интервалы будут расширяться примерно так:

$[0,0] \sqsubseteq [0,1] \sqsubseteq [0,7] \sqsubseteq [0,+\infty]$

Такой результат менее точный, чем полный интервальный анализ, но он безопасен и позволяет алгоритму завершиться.

## Анализ размера переменных. (слайд 28-29)

#### Какую решётку можно предложить для анализа размера одного значения?

Для одного абстрактного значения можно использовать конечную решётку размеров:

```text
        any
         |
       bigint
         |
        int
       /   \
    byte   char
       \   /
       bool
        |
        ⊥
```

Смысл элементов:

* $\bot$ — информации ещё нет или состояние недостижимо;
* $bool$ — значение помещается в $1$ бит;
* $byte$ — значение помещается в $8$-битное signed-число;
* $char$ — значение помещается в $16$-битное unsigned-число;
* $int$ — значение помещается в $32$-битное signed-число;
* $bigint$ — произвольное целое число;
* $any$ — произвольное значение, не обязательно целое.

Порядок можно понимать как отношение “помещается в”. Например:

$bool \sqsubseteq byte$

$bool \sqsubseteq char$

$byte \sqsubseteq int$

$char \sqsubseteq int$

$int \sqsubseteq bigint$

$bigint \sqsubseteq any$

Элементы $byte$ и $char$ несравнимы: $byte$ может хранить отрицательные значения, а $char$ может хранить значения больше $127$. Поэтому:

$byte \sqcup char = int$

#### Какие ещё решётки нужны для анализа целой программы?

Для анализа всей программы нужна решётка состояний:

$State = Vars \to Size$

где $Vars$ — множество переменных программы, а $Size$ — решётка размеров одного значения.

Состояние анализа сопоставляет каждой переменной её текущий абстрактный размер:

```text
x -> byte
y -> char
z -> int
```

Порядок, join и bottom задаются поточечно:

$\sigma_1 \sqsubseteq \sigma_2 \iff \forall x \in Vars:\ \sigma_1(x) \sqsubseteq \sigma_2(x)$

$(\sigma_1 \sqcup \sigma_2)(x) = \sigma_1(x) \sqcup \sigma_2(x)$

$\bot_{State}(x) = \bot$

Для flow-sensitive анализа также нужна решётка отображений из вершин CFG в состояния:

$CfgNode \to State$

Анализ является forward-анализом, потому что размер переменной после инструкции зависит от размеров значений до этой инструкции.

#### Как вычислять размеры выражений?

Для константы берём минимальный размер, в который она помещается:

$eval(n) = \min{T \mid n \in values(T)}$

Например:

$eval(1) = bool$

$eval(100) = byte$

$eval(1000) = char$

$eval(-1) = byte$

$eval(100000) = int$

Для переменной:

$eval(x, \sigma) = \sigma(x)$

Для арифметических операций можно вычислять результат через диапазоны значений соответствующих размеров и выбирать минимальный размер, который содержит все возможные результаты:

$eval(E_1\ op\ E_2, \sigma) = sizeOf(values(eval(E_1, \sigma))\ op\ values(eval(E_2, \sigma)))$

Например:

$byte \sqcup char = int$

поэтому выражение, в котором смешиваются $byte$ и $char$, обычно даёт результат размера $int$.

Для сравнений результат имеет размер $bool$:

$eval(E_1 < E_2, \sigma) = bool$

$eval(E_1 == E_2, \sigma) = bool$

Для приведения типа:

$eval((T)E, \sigma) = T$

Например:

$eval((byte)E, \sigma) = byte$

$eval((int)E, \sigma) = int$

#### Какие transfer-правила можно использовать?

Для объявления переменной:

$[[var\ x]](\sigma) = \sigma[x \mapsto \bot]$

Для присваивания:

$[[x = E]](\sigma) = \sigma[x \mapsto eval(E, \sigma)]$

Для остальных инструкций, которые не меняют значения переменных:

$[[n]](\sigma) = \sigma$

На слиянии веток используется join состояний. Если на одной ветке переменная имеет размер $byte$, а на другой $char$, то после слияния:

$byte \sqcup char = int$

#### Пример программы

```tip
var x, y, z, w;

if (input) {
    x = (byte) 10;
} else {
    x = (char) 1000;
}

y = (bool) 1;
z = x + y;
w = (bigint) z;
```

После ветвления для переменной $x$ получаем:

$x \mapsto byte \sqcup char = int$

Для $y$:

$y \mapsto bool$

Для $z = x + y$:

$z \mapsto int$

потому что $x$ имеет размер $int$, а $y$ имеет размер $bool$, и результат арифметической операции безопасно считать $int$.

Для $w = (bigint)z$:

$w \mapsto bigint$

Итоговое состояние анализа:

```text
x -> int
y -> bool
z -> int
w -> bigint
```
